# Development Log - Version 0.2

## Objective

Replace the self made sample log with a real sample Auth.log simulating attack attempts, replace conditionals to adapt to real SSH vocabulary and create a severity count to count the frequency ofsecurity-relevant event

---

## Development Process
---
I began by creating a with statement to automate the opening and closing of the sample Auth.log file. I then altered my conditional statements to include accurate SSH vocabulary from the log, identifying key events such as "Failed password", "error", and "Disconnecting". These events were possible signs of an attack attempt. Multiple failed passwords could highlight a possible brute force attempt, whereas the "error" and "Disconnecting" events were linked as cause and effect: the "error" line records the maximum authentication attempts being exceeded, and the "Disconnecting" line is the server's response, dropping the connection after too many failures. Identifying these in the conditional statements was crucial as they highlight real-world security concerns a SOC analyst would triage.
---

## Challenges

The challenges in this version were smaller and more conceptual than v0.1, which reflected a more solid foundation going in.
The main issue was file handling. The program initially couldn't locate the log file, returning a FileNotFoundError. This came down to two things: the relative path didn't correctly point to where the file lived in the folder structure, and the filename contained spaces, which caused problems. I resolved this by reading the error, checking the file's real name and location, correcting the path, and renaming the file to remove spaces. The lesson carried forward is to avoid spaces in filenames and to be precise with relative paths.
The second challenge was less a bug and more a realisation: my v0.1 conditionals were built around the words "ERROR" and "WARNING", but real SSH auth logs don't use that vocabulary. Applying the old conditionals to the new log would have matched almost nothing. This taught me that real-world log parsing means reading the actual format and adapting to it, rather than assuming the vocabulary a tutorial might use.
Smaller snags included briefly confusing a standalone variable with a dictionary key when writing the increment logic, and an indentation slip that placed a conditional outside the loop. Both were quickly identified and fixed.

---

## Skills Developed

Through this version of the project I gained experience with:

Safe file handling — using a with statement to automatically open and close files, rather than manually managing the file connection.

Dictionaries for data aggregation — using a dictionary to store and increment counts across multiple event types, replacing the repetitive conditional blocks from v0.1.

Real-world log analysis — reading an actual SSH auth log, identifying security-relevant events, and understanding what they indicate (brute force, authentication failures, forced disconnects).

Adapting code to real data — recognising that real logs use different vocabulary from assumptions, and adjusting conditionals to match the actual format and casing.

Debugging file paths — resolving relative path and filename issues by reading error output and correcting systematically.

Defensive security mindset — framing the tool around what a SOC analyst would triage, not just what the code does.
---

## Outcome

Version 0.2 successfully parses a realistic SSH auth log of ~6,900 lines and produces a summary count of security-relevant events, rather than printing every matching line as v0.1 did. Running against the sample log, it identified 189 authentication errors, 713 failed password attempts, and 189 forced disconnects — surfacing clear signs of brute-force activity that would be impractical to spot by scrolling the raw log.
The parser now demonstrates the core function of a basic log-triage tool: taking noisy, real-world log data and reducing it to an actionable summary.
